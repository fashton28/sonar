The vocabulary

The file map

tRPC's setup feels scattered the first time. It isn't — each file owns exactly one concern. Read it as three groups:

① Define the API (server brain)

src/trpc/init.ts
// The factory. Creates `t`, then exports the building blocks:
createTRPCRouter   // = t.router
baseProcedure      // = t.procedure
createTRPCContext  // builds ctx per request  (today: fake { userId })
src/trpc/routers/_app.ts
// The registry. Imports the builders, declares procedures, exports:
export const appRouter = createTRPCRouter({ /* procedures */ });
export type AppRouter = typeof appRouter;   // ← the magic type

② Expose the API (transport)

src/app/api/trpc/[trpc]/route.ts
// HTTP front door. fetchRequestHandler routes /api/trpc/* into appRouter.
export { handler as GET, handler as POST };

The one idea that unlocks it all

The router type flows to the client, the router code never does. import type { AppRouter } is erased at build time — your server logic stays on the server, but its shape (every input & output type) is known to the client. That's "end-to-end type safety with no codegen."

Procedure-builder cheat snippet

// A query (read):
list: baseProcedure
  .input(z.object({ limit: z.number().default(20) }))   // validate input
  .query(({ input, ctx }) => { /* return data */ });

// A mutation (write):
create: baseProcedure
  .input(z.object({ text: z.string().min(1) }))
  .mutation(async ({ input, ctx }) => { /* write, return */ });

Two arguments reach every procedure: input (validated) and ctx (per-request). There is no Express req/res — that's the adapter's job.
