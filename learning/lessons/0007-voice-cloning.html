Voice Cloning

From "the default voice" to "your voice." This is the lesson where Sonar grows a memory — and you'll see why a single feature drags in storage, a new router, and the model schema that was waiting for it all along.

The win for this lesson

A complete, build-it-yourself blueprint: the architecture, the one key decision, and code shapes for every piece — Modal, R2, tRPC, client. You're driving the implementation; this is the map and the hard parts called out, not a copy-paste dump.

How Chatterbox clones a voice

It's almost anticlimactic. You already deployed the model that does this. Cloning is just one extra argument to the same call:

wav = model.generate(text, audio_prompt_path="reference_voice.wav")

Zero-shot — no training, no fine-tuning. The model conditions on a short reference clip at inference time. Quality rules of thumb (Resemble's docs): a clean ~10-second clip, single speaker, no background noise, 24 kHz+, and a speaking style close to what you want out. Garbage reference → garbage clone.

The architectural shift — say it out loud

v1 was stateless: text in, audio out, nothing remembered. A "voice" is the opposite — it's a saved, reusable thing you upload once and pick many times. The moment you need to keep something between requests, you need storage. That single realization is what pulls in R2, a voices router, and the Voice table. Voice cloning isn't a model trick; it's a data feature.

The payoff you already set up

Open prisma/schema.prisma. The Voice model is already there — orgId, name, variant (SYSTEM | CUSTOM), r2ObjectKey. Past-you designed exactly this. No migration needed. Same for the prisma handle we put in ctx back in lesson 0003 — you're about to cash both in.

The two flows

CLONE (upload once)
  file picker → voices.create (tRPC) → store .wav in R2 → Voice row { orgId, r2ObjectKey, CUSTOM }

GENERATE WITH A VOICE (reuse many times)
  pick voiceId → tts.generate (tRPC)
                   → look up Voice (org-scoped) → presigned GET url for its r2ObjectKey
                   → POST { text, params, voice_url } to Modal
                        → container downloads the ref, passes audio_prompt_path → clone
                   → audio back → play

The ONE decision: how does the reference reach the GPU?

The model needs the reference as a local file path in the container. The clip lives in R2. Bridging that gap is the whole design choice:

OptionHowVerdict
A. Send bytes every requestbase64 the ref into each generate POSTSimplest, but re-uploads the clip on every generation. Wasteful for a voice you reuse.
B. Modal Volume of voicesPre-load wavs into a Volume, reference by nameEfficient, but needs a separate upload-to-Volume pipeline and name bookkeeping.
C. R2 + presigned URL ★Store in R2; pass a short-lived GET url; container downloads & cachesRecommended. Decouples storage from compute, reuses the R2 you need anyway, container caches per voice.

We go with C. It mirrors the v1 seam philosophy: the model stays a dumb service that takes a URL; your tRPC layer owns storage, auth, and tenancy.

The hard parts (shapes, not full files)

① Modal: download + cache the reference

modal/sonar_tts.py
import hashlib, os, urllib.request
from urllib.parse import urlparse

@modal.method()
def generate(self, text, exaggeration=0.5, cfg_weight=0.5, temperature=0.8, voice_url=None):
    audio_prompt_path = self._fetch_voice(voice_url) if voice_url else None
    wav = self.model.generate(
        text, audio_prompt_path=audio_prompt_path,
        exaggeration=exaggeration, cfg_weight=cfg_weight, temperature=temperature,
    )
    # …encode to wav bytes as before…

def _fetch_voice(self, url):
    # Cache by the OBJECT KEY (url path), NOT the full url — presigned urls change
    # every request (new signature), so caching by full url would never hit.
    key = hashlib.sha256(urlparse(url).path.encode()).hexdigest()
    path = f"/tmp/voices/{key}.wav"
    if not os.path.exists(path):
        os.makedirs("/tmp/voices", exist_ok=True)
        urllib.request.urlretrieve(url, path)   # download the ref into the container
    return path

Then have api(item) read item.get("voice_url") and pass it through. Redeploy. The cache means a hot voice is downloaded once per warm container.

② R2 client + presigned helpers

src/lib/r2.ts
import { S3Client, PutObjectCommand, GetObjectCommand } from "@aws-sdk/client-s3";
import { getSignedUrl } from "@aws-sdk/s3-request-presigner";
import { env } from "./env";

const r2 = new S3Client({
  region: "auto",
  endpoint: `https://${env.R2_ACCOUNT_ID}.r2.cloudflarestorage.com`,
  credentials: {
    accessKeyId: env.R2_ACCESS_KEY_ID,
    secretAccessKey: env.R2_SECRET_ACCESS_KEY,
  },
});

export async function r2Put(key: string, body: Buffer, contentType: string) {
  await r2.send(new PutObjectCommand({
    Bucket: env.R2_BUCKET_NAME, Key: key, Body: body, ContentType: contentType,
  }));
}

export function r2PresignedGet(key: string, expiresIn = 300) {
  return getSignedUrl(r2, new GetObjectCommand({
    Bucket: env.R2_BUCKET_NAME, Key: key,
  }), { expiresIn });   // short TTL: just long enough for Modal to download
}

R2 speaks the S3 API, so the AWS SDK "just works" once you point endpoint at your account's R2 host (Cloudflare presigned-URL docs). Get the four creds from the R2 dashboard → "Manage R2 API Tokens".

③ The voices router (cash in ctx.prisma + ctx.orgId)

src/trpc/routers/voices.ts
export const voicesRouter = createTRPCRouter({
  list: protectedProcedure.query(({ ctx }) =>
    ctx.prisma.voice.findMany({
      where: { orgId: ctx.orgId },              // ← tenancy, guaranteed by middleware
      orderBy: { createdAt: "desc" },
    }),
  ),

  create: protectedProcedure
    .input(z.object({ name: z.string().min(1), audioBase64: z.string() }))
    .mutation(async ({ ctx, input }) => {
      const key = `voices/${ctx.orgId}/${crypto.randomUUID()}.wav`;
      await r2Put(key, Buffer.from(input.audioBase64, "base64"), "audio/wav");
      return ctx.prisma.voice.create({
        data: { orgId: ctx.orgId, name: input.name, variant: "CUSTOM", r2ObjectKey: key },
      });
    }),
});

④ Generate-with-a-voice

src/trpc/routers/tts.ts (add to generateInput + resolver)
// input: add  voiceId: z.string().optional()

let voice_url: string | undefined;
if (input.voiceId) {
  const voice = await ctx.prisma.voice.findFirst({
    where: { id: input.voiceId, orgId: ctx.orgId },   // ← scope! never trust the id alone
  });
  if (!voice?.r2ObjectKey) throw new TRPCError({ code: "NOT_FOUND" });
  voice_url = await r2PresignedGet(voice.r2ObjectKey);
}
// include voice_url in the JSON body you POST to Modal
Security reflex you should feel by now

The lookup filters by { id, orgId: ctx.orgId } — not just id. Without the orgId clause, someone could pass another org's voiceId and clone a voice they don't own. protectedProcedure hands you a trustworthy ctx.orgId precisely so every query can be tenant-scoped. Make it a habit: every where on org data carries orgId.

Add voiceId back to the form schema (optional) and feed the selected id into tts.generate. Empty → default voice; set → your clone.

Gotchas to design around

GotchaDesign response
Bad reference audio (noise, too short)Validate on upload (duration/type); show guidance ("~10s, clean, one speaker").
Presigned URL leaks / lives too longShort expiresIn (minutes). It only needs to survive one Modal download.
Big base64 upload through the serverOK for ~10s clips. Scale path: presigned PUT so the browser uploads straight to R2 (server never holds the bytes).
Cross-tenant accessEvery where includes orgId: ctx.orgId. No exceptions.
Cold container re-downloads refCache by object key in /tmp (shape ①). Hot voice = one download per container.

Check yourself

Q1Why does adding voice cloning force you to introduce storage (R2)?
v1 was stateless. A voice is uploaded once and reused many times — persistent state — and persistence means storage. The feature is really a data feature.
Q2Why cache the downloaded reference by object key, not the full presigned URL?
Each presigned URL carries a fresh signature/expiry, so the full URL differs every time. The stable identity is the object key (the URL path) — cache on that.
Q3The voice lookup filters by { id, orgId: ctx.orgId }. Drop the orgId — what breaks?
A unique id is not an authorization check. Without the orgId clause, passing a known/guessed id from another tenant would resolve their voice. Tenancy lives in the where.
