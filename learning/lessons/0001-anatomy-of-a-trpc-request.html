The Anatomy of a tRPC Request

You're sitting in _app.ts thinking "this is where the backend lives." It isn't — and seeing why is the thing that makes the whole stack click.

The win for this lesson

By the end you'll be able to trace a single button-click in Sonar all the way to Postgres and back, and name which file does each job. Once you own this map, writing procedures is just filling in blanks. We write zero code today — we build the model the code hangs off.

1. The misconception to kill first

Coming from Express or a REST mindset, it's natural to think one file holds "the backend." In tRPC the work is deliberately split across three concerns, each in its own file. Your routers/_app.ts is only the registry — the table of contents. The actual machinery is spread out on purpose, because each piece changes for different reasons.

ConcernQuestion it answersFile in Sonar
DefineWhat does each endpoint do? What's shared per request?init.ts + routers/_app.ts
ExposeHow does an HTTP request reach a procedure?app/api/trpc/[trpc]/route.ts
ConsumeHow do React components call it, type-safely?client.tsx (client) · server.tsx (RSC)

Hold that three-way split in your head. Everything below is just those three concerns in motion. The full term-by-term table lives in your tRPC Glossary & File Map — keep it open.

2. The lifecycle, end to end

Imagine the user types text in Sonar and the frontend calls a generations.create procedure (which you'll build soon). Here is the entire journey — memorize the shape, not the words:

  1. Component calls a hook. A "use client" component uses useTRPC() from client.tsx and hands the options to React Query (useMutation/useQuery). You call it like a typed function — autocomplete on inputs, because the AppRouter type is in scope.
  2. The link serializes & batches. httpBatchLink turns the call into an HTTP request to /api/trpc/generations.create. If several calls fire in the same tick, it batches them into one round-trip.
  3. The route handler receives it. route.ts exports GET/POST; fetchRequestHandler matches the procedure name against appRouter.
  4. Context is built. createTRPCContext() runs now, fresh for this request — today it returns a fake { userId: 'user_123' }; soon it'll hold the real Clerk user + the Prisma client + the org id.
  5. Input is validated. The procedure's Zod .input() schema parses the payload. Invalid → a BAD_REQUEST error before your logic ever runs.
  6. Your resolver runs. The .query()/.mutation() function executes with { input, ctx } — talks to Prisma, returns a plain object.
  7. The result travels home typed. Serialized back over HTTP; React Query caches it; your component re-renders. The return type is inferred end-to-end — no codegen, no manual interfaces.
Notice what's missing

No req/res inside your procedure. No route strings you type by hand. No response DTO you keep in sync. tRPC's trade: you give up REST's framework-agnosticism, you get a function call that happens to cross the network, fully typed.

3. Why the type safety is "free"

The single line at the bottom of routers/_app.ts is the keystone:

src/trpc/routers/_app.ts
export type AppRouter = typeof appRouter;

The client does import type { AppRouter }. Because it's a type import, it's erased at build time — none of your server logic, secrets, or Prisma calls ship to the browser. But the shape — every procedure name, every input, every return type — is known to TypeScript on the client. Change a procedure's return on the server, and the client red-squiggles instantly. That's the whole magic, and it costs one line.

4. Check yourself

Retrieval beats re-reading. Answer before you click — the struggle is the point.

Q1In Sonar, which file builds the ctx object that every procedure receives?
Context is defined in init.ts and invoked per request by the route handler. The router only lists procedures; the handler calls the context factory but doesn't define it.
Q2What actually crosses from server to client to give end-to-end type safety?
It's a pure import type. The runtime code never ships; the shape does. No codegen exists in tRPC — inference does all the work.
Q3Inside a procedure resolver, what two things are handed to you?
Procedures receive { input, ctx }. The adapter handles HTTP for you, so there's no req/res in your resolver — that's the point of the abstraction.

5. Look at your own code with new eyes

Before the next lesson, open these two files and read them as the Define concern. You'll recognize every line now:

  • src/trpc/init.ts — find createTRPCContext (step 4) and baseProcedure (the builder every procedure starts from).
  • src/trpc/routers/_app.ts — the hello procedure is a complete miniature of the lifecycle: .input() (step 5) → .query() (step 6).
One smell to notice

createTRPCContext currently returns { userId: 'user_123' } — a hardcoded lie. Sonar is org-scoped through Clerk, so every real procedure needs the true user and org here. That fake line is exactly what Lesson 0002 replaces. Sit with the question: where should the Prisma client and Clerk auth() enter the picture?
