Voice Cloning v1 — Bytes in the Request

The fastest possible cloning loop: upload a clip, type text, hear it back in that voice — with zero storage. The reference audio rides inside the request and is thrown away after. Prove the loop first; add R2 later.

The win for this lesson

You'll know exactly the three changes that turn the default-voice pipeline into a cloning pipeline — and precisely where each plugs into the UI scaffold I've built for you. The UI is done; you wire the backend (Modal line + tRPC mutation + the file→base64 step).

Why "no storage" is the right first move

Lesson 0007 showed the full, production cloning design — R2, a voices table, presigned URLs. That's a lot of moving parts to debug at once. v1 strips it to the bone: if the reference clip travels with the generate request as base64, you need no R2, no Voice row, no presigned URLs. You're testing one thing — "does cloning work?" — without dragging storage bugs into the same session. Once the loop is proven, bolting on persistence is a clean, separate step.

The loop

[upload pane] + [text]  →  generate()
   → file → base64 (browser)
   → cloning.generate (tRPC):  { text, audioBase64 }
        → POST Modal { text, voice_b64 }
             → decode → /tmp/ref.wav → model.generate(text, audio_prompt_path="/tmp/ref.wav")
        → wav bytes → base64 data URL
   → <AudioPlayer> plays the clone

Piece 1 — Modal: the "extra line" (really ~4)

You were right that the model side is trivial — Chatterbox clones by taking one extra argument (audio_prompt_path). The only work is getting the base64 onto disk as a temp file:

modal/sonar_tts.py
import base64, tempfile

@modal.method()
def generate(self, text, exaggeration=0.5, cfg_weight=0.5, temperature=0.8, voice_b64=None):
    audio_prompt_path = None
    if voice_b64:
        data = base64.b64decode(voice_b64)
        tmp = tempfile.NamedTemporaryFile(suffix=".wav", delete=False)
        tmp.write(data); tmp.flush()
        audio_prompt_path = tmp.name          # ← the clone happens because of this
    wav = self.model.generate(
        text, audio_prompt_path=audio_prompt_path,   # None → default voice; set → clone
        exaggeration=exaggeration, cfg_weight=cfg_weight, temperature=temperature,
    )
    # …encode to wav bytes exactly as before…

# in api(item): pass  voice_b64=item.get("voice_b64")  through to generate.local(...)

Then modal deploy modal/sonar_tts.py. Note: audio_prompt_path=None is a no-op, so your existing default-voice TTS keeps working unchanged. One endpoint, two behaviors.

Piece 2 — tRPC: a cloning mutation

Almost identical to tts.generate, plus an audioBase64 input it forwards as voice_b64. Put it in a new cloning router (or extend tts) — your call. It's a protectedProcedure for the same reason: it spends GPU money.

src/trpc/routers/cloning.ts (sketch)
generate: protectedProcedure
  .input(z.object({
    text: z.string().min(1).max(5000),
    audioBase64: z.string().min(1),          // the reference clip
  }))
  .mutation(async ({ input }) => {
    const res = await fetch(env.MODAL_TTS_URL, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ text: input.text, voice_b64: input.audioBase64 }),
    });
    if (!res.ok) throw new TRPCError({ code: "INTERNAL_SERVER_ERROR", message: `TTS ${res.status}` });
    const bytes = await res.arrayBuffer();
    return { audio: `data:audio/wav;base64,${Buffer.from(bytes).toString("base64")}` };
  }),

Piece 3 — the browser: File → base64 (no Node Buffer here!)

This is the one client-side gotcha. On the server you'd reach for Buffer, but this runs in the browser. Use FileReader — and avoid btoa(String.fromCharCode(...)), which blows the call stack on large files.

Gotchas to expect

GotchaNote
Request payload sizeYou're sending the reference clip base64 and getting audio base64 back. A ~10s ref + result is a few MB — fine for v1, but it's why storage (URLs) is the eventual upgrade.
Reference quality~10s, clean, one speaker, low noise. Bad input → bad clone. Consider validating duration/type on upload later.
Cold startSame as before — first call after idle is slow while the model loads.
Nothing persistsBy design. Refresh = the clip and result are gone. That's what the storage phase fixes.

Check yourself

Q1Why does v1 need no R2 / no Voice table?
No persistence means no storage. The clip is base64'd into the one request, used once, and discarded. Storage only matters when a voice must be reused across requests — that's v2.
Q2Why use FileReader in the browser instead of Buffer.from(...)?
The conversion happens client-side, where there's no Node Buffer. FileReader.readAsDataURL gives a base64 data URL safely (no call-stack blowup on big files).
Q3You add cloning.generate but the client says it doesn't exist. Most likely cause?
A router only exists to the client once it's mounted in appRouter. The AppRouter type is the contract — unmounted means invisible. (Lesson 0001's lifecycle, again.)
