Your First Real Round-Trip

We build a live healthcheck: a procedure you write, called from a client component, rendered in your dashboard — and your Clerk name routed through your own backend on the way out and back.

The win for this lesson

By the end you'll have a real page at /test that pings your tRPC API and shows the result. You'll have written a procedure, consumed it with useTRPC(), and seen end-to-end type safety with your own eyes. This is the lifecycle from Lesson 0001 — but running.

We touch four files. Two you'll edit, two you'll create. Build each step, then run it. Don't batch — the point is to see each piece light up.

Step A — Write the procedure (the "Define" concern)

Open src/trpc/routers/_app.ts. You already have hello. Add a sibling healthcheck query. A query takes no side effects and just reads — perfect for "is the backend alive?"

src/trpc/routers/_app.ts
export const appRouter = createTRPCRouter({
  hello: baseProcedure
    .input(z.object({ text: z.string() }))
    .query((opts) => {
      return { greeting: `hello ${opts.input.text}` };
    }),

  // NEW — a no-input probe that proves the pipe works
  healthcheck: baseProcedure.query(() => {
    return {
      status: "ok" as const,
      timestamp: new Date().toISOString(),
    };
  }),
});

export type AppRouter = typeof appRouter;
Why .toISOString() and not new Date()?

The default transport is plain JSON, which has no Date type — a raw Date would silently arrive as a string on the client while TypeScript still thinks it's a Date. Returning a string keeps the type honest. (Want real Dates across the wire? That's exactly what the commented-out superjson transformer in client.tsx / query-client.ts is for — a later lesson.)

That's the entire backend change. Notice: no route string, no response type written by hand. The AppRouter type just grew a healthcheck member, and your client already knows about it.

Step B — The client component (the "Consume" concern)

useTRPC() is a hook, so it only runs in a Client Component. Create a new file in your test route and mark it "use client".

src/app/(dashboard)/test/health-check.tsx
"use client";

import { useQuery } from "@tanstack/react-query";
import { useUser } from "@clerk/nextjs";
import { useTRPC } from "@/trpc/client";

export function HealthCheck() {
  const trpc = useTRPC();          // the typed proxy from client.tsx
  const { user } = useUser();      // Clerk, client-side

  // queryOptions() builds the {queryKey, queryFn} for React Query.
  const health = useQuery(trpc.healthcheck.queryOptions());

  // Feed the Clerk name through YOUR backend via the existing `hello`:
  const greeting = useQuery(
    trpc.hello.queryOptions({ text: user?.firstName ?? "stranger" }),
  );

  if (health.isLoading) return <p>Pinging backend…</p>;
  if (health.error) return <p>❌ {health.error.message}</p>;

  return (
    <div className="space-y-1 font-mono text-sm">
      <p>✅ status: {health.data.status}</p>
      <p>🕐 server time: {health.data.timestamp}</p>
      <p>👋 {greeting.data?.greeting}</p>
    </div>
  );
}
The two lines that matter most

trpc.healthcheck.queryOptions() — this is the bridge. The @trpc/tanstack-react-query integration gives every procedure a .queryOptions() (for queries) and .mutationOptions() (for mutations) that produce a ready-made object for React Query — including a query key derived from the procedure name + input. You never write query keys or fetch URLs by hand.

Type health.data. and watch autocomplete offer status and timestamp — typed straight from your server resolver. That squiggle-free autocomplete is the payoff from Lesson 0001's AppRouter type.

Step C — Mount it (the "use client" boundary in action)

The page itself stays a Server Component (no "use client") — it just renders the client island. This is the modularization pattern you already know: a simple page.tsx that drops in a focused subcomponent.

src/app/(dashboard)/test/page.tsx
import { HealthCheck } from "./health-check";

export default function TestPage() {
  return (
    <div className="p-8 space-y-4">
      <h1 className="text-xl font-semibold tracking-tight">
        tRPC Health Check
      </h1>
      <HealthCheck />
    </div>
  );
}

Because it lives under (dashboard), your sidebar layout wraps it for free — no extra wiring. The provider that makes useTRPC() work is already mounted up in app/layout.tsx (TRPCReactProvider), so the hook just works here.

Step D — Run it & read the feedback loop

  1. Start the dev server: npm run dev
  2. Visit http://localhost:3000/test (sign in / pick an org if prompted).
  3. You should see three lines: a green status: ok, a server timestamp, and hello <your first name>.
  4. Open the Network tab and refresh. Find the request to /api/trpc/... — you'll see healthcheck and hello batched into one HTTP call. That's httpBatchLink from Lesson 0001, live.

Notice the name may flash stranger then update to yours: useUser() hydrates a tick later, the input changes, the query key changes, and the hello query re-runs automatically. That's React Query reacting to a new key — worth pausing on.

If it doesn't work — diagnose, don't guess

SymptomMost likely cause
useTRPC must be used within… / hook errorMissing "use client" at the top of health-check.tsx, or the component isn't under TRPCReactProvider.
health.data is undefined in TSYou read data before the isLoading/error guards. tRPC types data as possibly-undefined until resolved — guard first (as we did).
404 on /api/trpc/healthcheckProcedure name typo, or you didn't save _app.ts. The name in the URL must match the key in the router.
Redirected to /org-selectionWorking as intended — your middleware forces an org. Pick one and continue.

Check yourself

Q1Why must HealthCheck be a Client Component?
useTRPC() + useQuery are hooks — they need React's client runtime and the provider. (Server Components call tRPC a different way, via the server.tsx proxy — a later lesson.)
Q2What does trpc.healthcheck.queryOptions() hand to useQuery?
It produces the { queryKey, queryFn } object React Query needs — the key comes from the procedure name + input, so you never hand-write keys or URLs.
Q3You return new Date() instead of .toISOString(). What breaks?
Plain JSON has no Date type, so it serializes to a string — but the inferred type stays Date, a silent lie. A transformer like SuperJSON fixes this; until then, return primitives.

What you just proved

The pipe is real and typed. But look closely at the 👋 line: the name only got to the server because the client sent it as input. Your backend still has no idea who's logged in — remember createTRPCContext returns a hardcoded userId: 'user_123'. A real app can't trust the client to say who it is.
