Your Model Is a Web Service

The Chatterbox model is Python-native and needs a GPU. Your app is TypeScript on Vercel. This lesson is about the one decision that makes those two facts stop fighting each other — and it's the most important architectural call in the whole project.

The win for this lesson

You'll be able to draw Sonar's TTS data flow end to end and defend every boundary — why the model is a separate service, why TypeScript never runs the model, why the browser never talks to Modal, and why HTTP (not a language bridge) is the seam. No code today; we commit to the shape the next lessons fill in.

The false choice: "Python or TypeScript?"

You felt the tension correctly: Chatterbox/Modal are Python-native (Modal's own docs say so), but Sonar is TypeScript. The instinct is to pick one — run Python somehow inside Node, or find a TS way to do inference. Both are traps.

Tempting but wrongWhy it hurts
Spawn Python from Node / bundle Python on VercelVercel has no GPU; you'd ship a fragile cross-language process bridge and a multi-GB runtime.
Do inference in TypeScriptThe model, its weights, and its ecosystem are Python + CUDA. You'd be reinventing a wheel that doesn't roll in JS.
Let the browser call Modal directlyLeaks your Modal URL + token to every user; no auth, no validation, no org-scoping, no persistence.

The resolution: don't bridge languages inside a process — bridge them over the network. Each side stays fully idiomatic, and they meet at exactly one place: an HTTP endpoint.

① The Modal endpoint — a "deep module"

A deep module is one with a tiny interface hiding huge complexity. Modal's Chatterbox service is the textbook case. Its interface is one line:

POST { prompt, <voice + params> }   →   audio/wav bytes

Behind that sliver, hidden completely from you: GPU allocation (an A10G), gigabytes of model weights, warm-loading the model once per container with @modal.enter, request concurrency, and autoscaling to zero when idle. You will summon all of it with a single fetch(). That asymmetry — trivial to call, mountainous inside — is exactly what good architecture buys you.

② The tRPC procedure — the translator (and the bouncer)

This is why the call routes through your server instead of browser→Modal. The procedure is the only layer that can safely own:

  • Secrets — the Modal URL and auth token live in server env, never shipped to the browser.
  • Auth + tenancy — it's a protectedProcedure (Lesson 0003): real user, guaranteed ctx.orgId.
  • Validation — Zod parses the text + params before a single GPU-second is spent.
  • Persistence — later, it writes the Generation row and stores the WAV in R2.

The browser stays dumb on purpose: it asks your API for audio and plays the result. It never learns that Modal exists.

③ The client — fire a mutation, play the result

Generating audio is a write (it costs money, creates a record), so it's a mutation, not a query. You already know the hook shape from Lesson 0002 — useMutation(trpc.tts.generate.mutationOptions()) — wired to your existing form's onSubmit.

The payoff of the seam: swappability

Because the boundary is plain HTTP, the model provider is a detail. Move from Modal to RunPod, or swap Chatterbox for another model, and only the inside of one tRPC function changes. The form, the client, the DB, the auth — untouched. A clean seam is an option you can exercise later for free.

The serverless-GPU reality (so you're not surprised)

Modal scales to zero. The first request after idle pays a cold start: a container boots and @modal.enter loads the model (this can take tens of seconds). Warm requests are fast. The example keeps containers alive with a scaledown_window (~5 min) and allows ~10 concurrent requests per container. Design implication for Sonar: your mutation may take a while on a cold path — the form's "Generating…" spinner (already built!) and a generous timeout matter. Modal web endpoints have "all the superpowers of Modal Functions" — GPUs, Secrets, Volumes, autoscaling.

Check yourself

Q1Why does the TTS call route through tRPC instead of the browser calling Modal directly?
The procedure is the bouncer: it holds the Modal URL + token, enforces auth and org-scoping, validates input, and persists results. Direct browser→Modal would leak the token and skip all of that.
Q2What makes the Modal endpoint a "deep module"?
One line in (text + params), audio out — but GPUs, model weights, warm-loading, and autoscaling hide behind it. Small surface, huge inside: the definition of a deep module.
Q3Generating speech is modeled as a tRPC mutation rather than a query because…
Queries read; mutations change things. Generation spends GPU money and creates a Generation row — a write with side effects — so it's a mutation.
