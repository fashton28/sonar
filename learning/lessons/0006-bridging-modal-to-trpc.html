Bridging Modal to tRPC

The complete TypeScript half: from a slider on the form to a cloud GPU and back to an <audio> tag playing in the browser. Every file, every gotcha.

The win for this lesson

A mental + literal map of the entire pipeline. After this you could implement it yourself — and you'll understand the two non-obvious decisions inside it: how binary audio crosses a JSON RPC boundary, and how your form's sliders map onto Chatterbox's real parameters.

The whole flow on one screen

TTS form  ──submit──▶  TextToSpeechForm.onSubmit
                            │  useMutation(trpc.tts.generate)
                            ▼
                       tRPC  tts.generate  (protectedProcedure, server)
                            │  1. Zod-validate text + params

                            ▼

Five files change. We'll go in dependency order: env → router → mount → form params → client playback.

① The endpoint URL is a server secret — env.ts

The Modal URL must never reach the browser (Lesson 0004's rule). It belongs in the server half of your validated env, so it's impossible to import client-side.

src/lib/env.ts
server: {
  DATABASE_URL: z.string().url(),
  CLERK_SECRET_KEY: z.string().min(1),
  APP_URL: z.string().min(1),
  MODAL_TTS_URL: z.string().url(),   // ← add this
},

Add MODAL_TTS_URL=https://…modal.run to your .env. Now if it's missing, the app refuses to boot — a loud failure beats a mysterious one at request time.

② The bridge — src/trpc/routers/tts.ts (new)

A new sub-router with one mutation. It's a protectedProcedure (Lesson 0003) — only real, org-scoped users may spend GPU money.

src/trpc/routers/tts.ts
import { z } from "zod";
import { TRPCError } from "@trpc/server";
import { createTRPCRouter, protectedProcedure } from "../init";
import { env } from "@/lib/env";

const generateInput = z.object({
  text: z.string().min(1).max(5000),
  exaggeration: z.number().min(0).max(1),
  cfgWeight: z.number().min(0).max(1),
  temperature: z.number().min(0).max(2),
});

export const ttsRouter = createTRPCRouter({
  generate: protectedProcedure
    .input(generateInput)
    .mutation(async ({ input }) => {
      const res = await fetch(env.MODAL_TTS_URL, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({
          text: input.text,
          exaggeration: input.exaggeration,
          cfg_weight: input.cfgWeight,      // camelCase → snake_case at the boundary
          temperature: input.temperature,
        }),
      });

      if (!res.ok) {
        throw new TRPCError({
          code: "INTERNAL_SERVER_ERROR",
          message: `TTS service responded ${res.status}`,
        });
      }

      // The subtle part — see callout below:
      const bytes = await res.arrayBuffer();
      const base64 = Buffer.from(bytes).toString("base64");
      return { audio: `data:audio/wav;base64,${base64}` };
    }),
});
Subtlety #1 — binary audio over a JSON boundary

tRPC speaks JSON. Modal hands back raw audio/wav bytes. Bytes aren't JSON. So we read them as an ArrayBuffer, base64-encode them into a data: URL, and return that string. The browser can drop a data URL straight into <audio src> — no Blob juggling. Trade-off: base64 is ~33% larger and the whole clip passes through your server's memory. Perfect for v1; the storage phase replaces it with "upload to R2, return a short URL" — which is exactly why we kept audio out of the database.

Subtlety #1b — the boundary is where names get translated

Your TS uses cfgWeight (camelCase, idiomatic JS); Python/Chatterbox wants cfg_weight (snake_case). The tRPC procedure is the natural place to translate. Each side stays idiomatic; the adapter reconciles them. This is what "adapter" layers are for.

③ Mount the sub-router — _app.ts

Routers nest. _app.ts stays a clean table of contents; the TTS logic lives in its own file.

src/trpc/routers/_app.ts
import { ttsRouter } from "./tts";

export const appRouter = createTRPCRouter({
  hello: /* … */,
  healthcheck: /* … */,
  me: /* … */,
  tts: ttsRouter,        // ← now the client has trpc.tts.generate
});

That one line gives the client a fully-typed trpc.tts.generate. This is how you keep a growing API tidy: one file per feature, mounted here.

④ Reconcile the form with the model's real knobs

Your form was scaffolded with temperature, topP, topK, repetitionPenalty — generic LLM params. Chatterbox doesn't use most of those. Its real knobs are exaggeration, cfg_weight, and temperature. We make the UI honest. Because the settings panel maps over the sliders array, changing one data file re-skins the whole panel.

What about the Generation Prisma model?

It still has topP/topK/repetitionPenalty columns. We're not persisting in v1, so leave the schema for now — we'll migrate those columns to the real params in the storage phase, alongside writing rows. One change at a time.

⑤ Client: fire the mutation, play the result

The form's onSubmit becomes the trigger. Beautifully, TanStack Form sets isSubmitting = true for the duration of an async onSubmit — so your existing "Generating…" spinner (the GenerateButton) just works, no extra wiring. We hold the resulting audio URL in a tiny context so the preview pane can read it.

src/features/text-to-speech/components/text-to-speech-form.tsx
"use client";
import { createContext, useContext, useState } from "react";
import { useMutation } from "@tanstack/react-query";
import { useTRPC } from "@/trpc/client";
/* …zod schema + defaults from step ④… */

const TTSResultContext = createContext<{ audioUrl: string | null }>({ audioUrl: null });
export const useTTSResult = () => useContext(TTSResultContext);

export function TextToSpeechForm({ children }: { children: React.ReactNode }) {
  const trpc = useTRPC();
  const [audioUrl, setAudioUrl] = useState<string | null>(null);
  const generate = useMutation(trpc.tts.generate.mutationOptions());

  const form = useAppForm({
    ...ttsFormOptions,
    defaultValues: defaultTTSValues,
    validators: { onSubmit: ttsFormSchema },
    onSubmit: async ({ value }) => {
      const { audio } = await generate.mutateAsync(value);  // ← the round trip
      setAudioUrl(audio);
    },
  });

  return (
    <TTSResultContext.Provider value={{ audioUrl }}>
      <form.AppForm>{children}</form.AppForm>
    </TTSResultContext.Provider>
  );
}

Why the existing spinner "just works"

You built GenerateButton to read form.state.isSubmitting. TanStack Form keeps that flag true while your async onSubmit awaits the mutation. So the moment you make onSubmit call mutateAsync, the spinner, the disabled state, and the textarea-disable you already wired all light up for free. Good architecture pays you back later.

The gotchas that bite in production

GotchaWhat to do
Cold-start timeout — first call can take 30–60s while Modal boots + loads the model.Fine locally. On Vercel, serverless functions have max durations (hobby ~10–60s) — long generations may need a higher limit, a warm container, or the storage/polling pattern later.
Payload size — base64 of a long clip can be several MB.OK for v1. The R2 phase returns a short URL instead, killing this entirely.
Open endpoint — anyone with the URL can spend your GPU.Add a bearer token: a Modal Secret the procedure sends as Authorization: Bearer …; the Python endpoint checks it.
Runtime — Buffer needs the Node runtime.tRPC route handlers are Node by default — good. Don't move this to the Edge runtime.

Check yourself

Q1Why does the mutation base64-encode the audio instead of returning raw bytes?
tRPC serializes to JSON; binary needs encoding to survive. base64 (slightly larger) rides inside JSON, and a data: URL plugs straight into <audio>. R2 later returns a plain URL instead.
Q2Why is generate a protectedProcedure, not a baseProcedure?
Generation costs money and (soon) writes org-scoped rows. The auth middleware guarantees a real user + ctx.orgId before any GPU-second is spent.
Q3You made onSubmit async and call mutateAsync. Why does the spinner work with no extra code?
GenerateButton reads form.state.isSubmitting, which TanStack Form keeps true for the lifetime of the async onSubmit. The spinner and disabled states you already built fire for free.

tRPC — Procedures & mutations and Modal — Web endpoints. For the data: URL trick, MDN's Data URLs.
