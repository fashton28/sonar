Real Context & Protected Procedures

We kill the userId: 'user_123' lie. By the end, your server knows who you are — and which org you're in — without the client telling it a thing. This is the line between a demo and a backend.

The win for this lesson

A new me query that returns your real identity, derived from the Clerk session on the server. To build it you'll learn the three load-bearing pieces of every serious tRPC backend: a real context, a middleware, and a protected procedure. Your new LSP squiggles will help you — lean on them.

Why this matters (the security idea)

In Lesson 0002 the greeting showed your name — but only because the client sent it as input. Anyone can send any input. If your "who am I" comes from the client, a user can claim to be anyone. Identity must be established on the server, from something the client can't forge — here, the Clerk session cookie. That's the whole reason we're rebuilding createTRPCContext.

Step A — Put real things in the context

Recall from the glossary: context is built fresh per request and handed to every procedure. It's the right home for two things every procedure needs — who's calling (Clerk) and how to reach the database (Prisma). Open src/trpc/init.ts and rewrite the top half:

src/trpc/init.ts
import { initTRPC, TRPCError } from "@trpc/server";
import { cache } from "react";
import { auth } from "@clerk/nextjs/server";
import { prisma } from "@/lib/db";

export const createTRPCContext = cache(async () => {
  const authState = await auth();          // ← reads the Clerk session
  return { auth: authState, prisma };      // ← available in every procedure
});

// Tell tRPC the SHAPE of ctx, so ctx.auth and ctx.prisma are typed everywhere:
export type Context = Awaited<ReturnType<typeof createTRPCContext>>;

const t = initTRPC.context<Context>().create();
The one new incantation: initTRPC.context<Context>()

Before, you had initTRPC.create() — tRPC assumed an empty ctx. Now you hand it the Context type, so inside any procedure, ctx.auth and ctx.prisma autocomplete and typecheck. Forget this line and every ctx.auth will squiggle — a perfect first job for your new diagnostics.

Notice auth() works here because Sonar's Clerk middleware (src/proxy.ts) runs on /api/trpc/* before this code, establishing the session. The cache() wrapper means if one request hits several procedures, auth() runs only once.

Step B — Gate access with a middleware

A middleware wraps a procedure: it runs first, decides whether to continue (next()), and can extend and refine the context. This is how auth, logging, and rate-limiting are done in tRPC. Add this below the t definition, then export a new procedure builder:

src/trpc/init.ts (continued)
export const createTRPCRouter = t.router;
export const createCallerFactory = t.createCallerFactory;
export const baseProcedure = t.procedure;

// Sonar is org-scoped: every real action needs BOTH a user and an org.
const isAuthed = t.middleware(({ ctx, next }) => {
  if (!ctx.auth.userId || !ctx.auth.orgId) {
    throw new TRPCError({ code: "UNAUTHORIZED" });
  }
  return next({
    ctx: {
      userId: ctx.auth.userId,   // was string | null  →  now string
      orgId: ctx.auth.orgId,     // was string | null  →  now string
    },
  });
});

export const protectedProcedure = t.procedure.use(isAuthed);
The middleware does TWO jobs — see both

1. It guards. No user or no org → UNAUTHORIZED, your resolver never runs.
2. It narrows types. By returning a new ctx from next(), any procedure built on protectedProcedure sees ctx.userId and ctx.orgId as plain string — never null. The guard you wrote at runtime becomes a guarantee at compile time. That two-for-one is the soul of tRPC middleware.

Why require orgId, not just userId?

Every row in Sonar's schema (Voice, Generation) is scoped by orgId, and proxy.ts already forces org selection. So a guaranteed ctx.orgId means every future query can safely filter where: { orgId: ctx.orgId } — the foundation of multi-tenant safety. We'll cash this in next lesson.

Step C — The me query (the payoff)

Open src/trpc/routers/_app.ts. First, import the new builder, then add a me query. auth() gave us IDs; to get the user's name we ask Clerk for the full profile with currentUser().

src/trpc/routers/_app.ts
import { z } from "zod";
import { currentUser } from "@clerk/nextjs/server";
import { baseProcedure, createTRPCRouter, protectedProcedure } from "../init";

export const appRouter = createTRPCRouter({
  // ...hello and healthcheck stay as they are...

  me: protectedProcedure.query(async ({ ctx }) => {
    const user = await currentUser();          // full Clerk profile
    return {
      userId: ctx.userId,                      // guaranteed string (middleware)
      orgId: ctx.orgId,                        // guaranteed string
      name: user?.firstName ?? "there",
    };
  }),
});
auth() vs currentUser() — know the difference

auth() is cheap: it reads IDs straight out of the session token, no network. Use it for gating and scoping (we did, in context). currentUser() makes a call to Clerk to fetch the full profile (name, email, image). Reach for it only when you need those fields. Don't put currentUser() in context — you'd pay for a profile fetch on every single procedure.

Check yourself

Q1How does the server know who's calling in the me query?
Identity is established server-side in createTRPCContext via auth(), reading the signed Clerk session. The client sends nothing — that's the entire point of this lesson.
Q2Besides throwing UNAUTHORIZED, what does the isAuthed middleware do?
By returning a refined ctx from next(), it turns userId/orgId from string | null into string for every protected procedure — a compile-time guarantee from a runtime check.
Q3Why use auth() in context but currentUser() in the resolver?
auth() reads ids from the token with no network cost — perfect for per-request context. currentUser() hits Clerk for the full profile, so you call it only where you actually need name/email.

What you just unlocked

Your context now carries a guaranteed orgId and a prisma handle. That's exactly the equipment needed to read real data safely scoped to the tenant. The healthcheck stays a baseProcedure (public — no login needed to ping). Everything that touches user or org data will be a protectedProcedure from here on.

Defense in depth — a pro habit

Sonar protects routes twice: proxy.ts guards at the routing layer, and protectedProcedure guards at the API layer. Belt and suspenders. Never assume your UI is the only caller — gate the procedure itself, every time.
