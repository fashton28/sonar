Build & Deploy the Modal Chatterbox App

We stand up the Python half of Sonar: a GPU-backed Chatterbox service with a public HTTPS URL. By the end you'll curl a sentence and get back a .wav generated on a cloud GPU.

The win for this lesson

A deployed Modal endpoint that turns text into speech. It's the "deep module" from Lesson 0004, now real. Your code lives in modal/sonar_tts.py — already written for you; this lesson explains every part and walks you through deploying it.

Step 1 — Read the file you'll deploy

Hard-won gotcha: don't name a script after a package it imports

This file is sonar_tts.py — not chatterbox.py (nor modal.py). A script whose name matches a package it imports shadows that package on sys.path: inside the container, from chatterbox.tts import … would resolve to your own file instead of the installed library and crash-loop the container (which shows up at the HTTP layer as a 303/timeout, not a clean error). We hit this live and renamed the file to fix it.

Open modal/sonar_tts.py. It's ~60 lines and has four ideas. Here's each, and why it's there.

① The Image — a reproducible container recipe

image = modal.Image.debian_slim(python_version="3.10").uv_pip_install(
    "chatterbox-tts==0.1.6",
    "fastapi[standard]==0.124.4",
    "peft==0.18.0",
)
with image.imports():
    import io, torchaudio as ta
    from chatterbox.tts import ChatterboxTTS
    from fastapi.responses import StreamingResponse

The Image describes the exact environment your code runs in — pinned to the same versions as Modal's official example so it builds first try. The with image.imports() block is the subtle part: those heavy imports (torch, chatterbox) only exist inside the container, not on your laptop. When the Modal CLI reads this file locally to figure out your app, it skips that block — so you don't need torch installed to deploy. Smart.

② The GPU class — load once, serve many

@app.cls(gpu="a10g", scaledown_window=60 * 5, timeout=60 * 10)
@modal.concurrent(max_inputs=10)
class Chatterbox:
    @modal.enter()
    def load(self):
        self.model = ChatterboxTTS.from_pretrained(device="cuda")
Why a class, and why @modal.enter?

Loading the model takes real time. You do not want that on every request. @modal.enter runs once per container, on cold start, and stashes the model on self. Every subsequent request reuses the warm model. This is the "cold start" you read about in 0004 made concrete: first call slow (boot + load), the rest fast. scaledown_window keeps the container warm 5 min after the last call, then it scales to zero so you stop paying.

③ The generation method

@modal.method()
def generate(self, text, exaggeration=0.5, cfg_weight=0.5, temperature=0.8) -> bytes:
    wav = self.model.generate(text, exaggeration=exaggeration,
                              cfg_weight=cfg_weight, temperature=temperature)
    buffer = io.BytesIO()
    ta.save(buffer, wav, self.model.sr, format="wav")
    buffer.seek(0)
    return buffer.read()

Plain Chatterbox: model.generate(text) uses a built-in default voice — which is exactly why we skipped the voice-cloning Volume the official example needed. We encode the waveform to WAV bytes in memory (no temp files) so it can travel over HTTP.

④ The HTTP seam

@modal.fastapi_endpoint(method="POST", docs=True)
def api(self, item: dict):
    audio_bytes = self.generate.local(item["text"], item.get("exaggeration", 0.5),
                                      item.get("cfg_weight", 0.5), item.get("temperature", 0.8))
    return StreamingResponse(io.BytesIO(audio_bytes), media_type="audio/wav")

This decorator is the entire bridge to your TypeScript world. It turns the method into a POST URL that accepts a JSON body (item: dict) and streams back audio/wav. self.generate.local(...) runs the generator in the same warm container. This URL is what your tRPC mutation will fetch() next lesson.

Step 2 — Deploy it

  1. Live-reload dev mode (great while iterating): modal serve modal/sonar_tts.py
  2. Or deploy a stable URL: modal deploy modal/sonar_tts.py

Modal builds the image (first time: a few minutes — it's installing torch + the model deps), then prints a URL ending in …modal.run for the api endpoint. Copy that URL.

Step 3 — Prove it works

curl -X POST <YOUR-API-URL> \
  -H 'Content-Type: application/json' \
  --data '{"text":"Sonar online. The future sounds like this."}' \
  --output /tmp/sonar.wav

open /tmp/sonar.wav   # plays it on macOS

First call is slow (cold start: container boot + model load — tens of seconds). Run it again and it'll be quick — that's the warm container. Hearing that sentence is today's milestone: a cloud GPU just spoke for you.

If the first deploy errors

Gated model / 401 from HuggingFace: standard Chatterbox is public, but if you hit an auth error, create a token at huggingface.co, then modal secret create hf-token HF_TOKEN=hf_xxx and add secrets=[modal.Secret.from_name("hf-token")] to @app.cls(...).
Missing dependency at runtime: read the traceback — it names the package; add it to uv_pip_install. (Classic runtime bucket — the deploy parsed fine, it broke while executing in the container.)

Security note — don't forget this

This v1 endpoint is open to anyone with the URL. Fine for today's local dev, but an open GPU endpoint is a billing risk. Before this lives anywhere real, we'll add a bearer token (a Modal Secret the tRPC procedure sends in an Authorization header). Flagged now so it doesn't get forgotten.

Check yourself

Q1Why is the model loaded in an @modal.enter method instead of inside generate?
@modal.enter runs once on cold start; the loaded model is reused across all requests to that warm container. Loading per-request would make every call pay the slow cold-start cost.
Q2What does with image.imports() accomplish?
torch/chatterbox only exist in the built container. The block keeps those imports out of the local parse step, so you can deploy without installing them on your machine.
Q3Your first curl takes 40 seconds, the second takes 2. Why?
First call = container boot + @modal.enter model load. After that the container is warm (kept alive by scaledown_window), so generation is fast.
